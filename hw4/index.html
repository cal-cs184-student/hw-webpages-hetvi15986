Overview

In this project, I built a real-time cloth simulation system that combines physics, collision handling, and GPU shading. First, I model the cloth as a grid of point masses connected by structural, shearing, and bending springs, with support for pinned points that allow for visualization. I simulated motion using Verlet integration, applying forces such as gravity and spring forces (and later wind forces), and used a constraint [Provot] to prevent excessive stretching under forces. Next, I implemented collisions with spheres and planes, correcting positions using recorded previous states and adding friction for realism. I also handled self-collision efficiently using spatial hashing to avoid expensive pairwise checks. To render, I implemented shaders, including diffuse and Blinn-Phong lighting, texture mapping, bump and displacement mapping, and environment reflections. I also created a custom shader combining edge detection with a Fresnel-based rim glow. For extra features, I added interactive controls like a movable sphere, adjustable wind forces, and a box collision primitive. Overall, this project helped me understand how physics simulation, collision systems, and GPU shading work together in real-time graphics applications, which I can then apply to our larger group project.

Part 3: Handling collisions with other objects

Deliverable: Describe your implementation of handling collisions with spheres and planes.

To handle collisions with spheres, I implemented Sphere::collide(PointMass &pm) as follows: I first compute the vector from the sphere’s center to the point mass and its length. If this distance is less than the sphere’s radius, the point mass is inside the sphere.

I then compute the tangent point on the sphere’s surface by projecting the point mass radially outward: 

tangent = origin + (pm.position - origin).unit() * radius;

The correction vector is: 

correction = tangent - pm.last_position;

The new position is updated using friction: 

pm.position = pm.last_position + (1 - friction) * correction;

This moves the point mass to the surface while reducing velocity to simulate energy loss.

For plane collisions, I compute the signed distance: 

d = dot(pm.position - plane.point, plane.normal);

If d < 0, the point mass has crossed the plane. I project it back: 

tangent = pm.position - d * normal;

To avoid numerical sticking, I offset slightly along the normal: 

target = tangent + SURFACE_OFFSET * normal;

Then apply the same friction-based correction: 

pm.position = pm.last_position + (1 - friction) * (target - pm.last_position);

Both methods rely on last_position to preserve the velocity approximation inherent in Verlet integration.

These collision checks are integrated into Cloth::simulate after Verlet integration but before the Provot constraint. For each timestep, I loop over all point masses and collision objects. This ensures collisions are resolved before spring corrections, preventing the cloth from being pulled through surfaces.

Part 4: Handling self-collisions

Deliverable: Describe your implementation of handling self-collisions.

To handle self-collisions efficiently, I implemented spatial hashing instead of an expensive O(n²) pairwise check. The algorithm consists of three parts:

1. Hash Function (Cloth::hash_position)

I partition the 3D space into boxes with dimensions: 

w = 3 * width / num_width_points h = 3 * height / num_height_points t = max(w, h)

For each point mass, I compute its box index: 

ix = floor(pos.x / w) iy = floor(pos.y / h) iz = floor(pos.z / t)

These indices are combined into a unique key: 

key = ix * 1000000 + iy * 1000 + iz

This allows point masses in nearby regions to be grouped efficiently.

2. Building the Spatial Map (Cloth::build_spatial_map)

At each timestep, I clear the existing map: 

unordered_map<float, vector<PointMass*>*>

Then I iterate through all point masses:

  • Compute the hash key
  • Insert the point mass pointer into the corresponding bucket
  • Create a new bucket if needed

This builds a fast lookup structure for nearby collision candidates.

3. Self-Collision Resolution (Cloth::self_collide)

For each point mass:

  • Retrieve candidates from the same spatial bucket
  • Skip itself
  • Check distance against neighbors

If the distance is less than 2 * thickness, a correction is applied: 

correction = direction_to_separate * (2 * thickness - distance) * 0.5

All corrections are accumulated: 

total_correction += correction neighbors += 1

Then averaged and scaled: 

pm.position += (total_correction / neighbors) / simulation_steps

This prevents instability from large sudden corrections.

Integration into Cloth::simulate

The simulation loop is structured as follows:

  1. Verlet integration
  2. External collisions (sphere, plane)
  3. build_spatial_map()
  4. Self-collision via self_collide()
  5. Provot constraint enforcement

This ordering ensures that collisions are resolved before spring corrections, preventing the cloth from being pulled back into invalid states.

This spatial hashing approach reduces complexity from O(n²) to near O(n), making real-time simulation practical. The chosen box size (3× scaling) balances accuracy and performance by controlling bucket density.

Deliverable: Explain in your own words what is a shader program and how vertex and fragment shaders work together to create lighting and material effects.
A shader program runs on the GPU to control rendering. The vertex shader processes each vertex. it transforms positions to screen space and passes along attributes. The fragment shader then runs per pixel. It receives interpolated attributes and uses them (with light data, textures, etc.) to compute the final color. Together, they turn raw geometry into lit, textured images.

For my custom shader, I implemented a stylized post‑processing effect that combines edge detection with a cyan‑blue rim glow. First, I compute standard Blinn‑Phong lighting to obtain a base color. Then, I apply screen‑space edge detection by taking the luminance of the base color and using fwidth to measure its rate of change; areas with high change are mixed toward black, producing dark outlines that emphasize the cloth’s silhouette and wrinkles. Next, I add a constant fresnel‑based glow: I compute (1 - |dot(N, V)|)^2 (where N is the normal and V is the view direction) and multiply by a fixed cyan‑blue color and a constant intensity (no time pulsation). This creates a bright rim effect around the cloth’s edges without any animation. The final output is the edge‑enhanced color plus the additive glow, resulting in a cartoon‑like, neon‑lit appearance that makes the cloth stand out while maintaining real‑time performance.

I added a spatially varying wind force to the cloth simulation to simulate realistic fluttering. The wind force is defined as `F_wind = wind_dir * wind_strength * sin(pos.x * wind_variation) * cos(pos.z * wind_variation)`, applied to each point mass (except pinned ones) after gravity but before spring forces. The wind direction, strength, and spatial frequency are adjustable via GUI sliders. Without wind, the cloth falls straight down and settles into a static draped shape with no motion. With wind enabled (e.g., strength = 5, variation = 0.8, direction diagonal), the cloth exhibits continuous waving and rippling, with waves propagating across the surface and the fabric lifting and twisting in mid‑air before landing. The spatial variation creates realistic asymmetric fluttering, unlike a uniform (constant) force which would simply push the cloth sideways.
